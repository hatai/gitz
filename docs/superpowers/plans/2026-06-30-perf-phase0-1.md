## Global Constraints

- `zig build test --summary all`（**Debug 既定維持**・実行時安全チェックを保つ）。**lint/format/typecheck/migration は存在しない**（`zig build test` が型検査も兼ねる）。
- benchmark は `zig build bench`（Debug 既定: correctness + alloc 数）・`zig build bench -Doptimize=ReleaseFast`（fps/latency）。`zig build test` は Debug のまま。
- **Zig 0.16 実 API（codex B1/B2）**: 計測は `std.Io.Clock.now(.awake, io)`（`init.io` 経由）の timestamp 差分。env 取得は `init.environ_map.get(...)`。`std.time.Timer`/`std.process.getEnvVar` は**存在しない**（`docs/superpowers/plans/zigzag-api-notes.md` が正）。
- 新規 `.zig`（`src/bench.zig`・`src/config.zig`）は `src/root_test.zig` へ import 追加（補助純粋関数のテストを書く場合）。
- 所有権規約: Msg/AppCmd ペイロードは複製所有・消費者が deinit（**Phase 1 はこの規約を不改変**・move semantics は Phase 2）。
- `std.ArrayList(T)` は unmanaged（`.empty` / `append(a, x)` / `toOwnedSlice(a)`）。OOM 教訓: `toOwnedSlice` 後は元 ArrayList の errdefer が無害化するため新 slice 用 errdefer を再登録。
- commit メッセージ規約: review ID（spec 節/codex ID）は commit subject 末尾へ括弧付き（例 `(§5.1/perf phase1)`）。
- **実行開始時に `main` から feature ブランチ `feat/perf-phase0-1` を切る**（#2/#1 と同じ no-ff flow）。
- 10万コミット前提だが CI で生成は重い→bench は手動実行・`bench/report-before.md` のみ commit（`bench/repos/` は git-ignore）。

## Task 5: model.zig — appendLogCommits move（codex M6 手順固定）

**Files:** Modify: `src/model.zig:280-300`・Test: `src/model.zig`

**設計（spec §5.1・codex M6）:** `appendLogCommits` の既存 items 再 dup を廃止。具体手順:
1. **新規分だけ先行 clone**（OOM で既存無傷）: `var appended: std.ArrayList(...) = .empty;` へ `ensureTotalCapacity(a, new_entries.len)` + `cloneCommit` で append。errdefer で appended を解放。
2. 既存 `self.log_commits` への統合: `try self.log_commits.ensureTotalCapacity(a, old_len + appended.items.len)`（OOM は appended errdefer が新規 clone 解放・既存は無傷）。
3. 成功後のみ `self.log_commits.appendSliceAssumeCapacity(appended.items)`（`@memcpy` による浅い copy・`log.Commit` の所有 ptr を複製）。
4. **即座に disarm**: `appended.items.len = 0;` → `appended.deinit(a)`（要素解放なし・ArrayList 構造のみ解放）。これで `appended` の errdefer/deinit が子要素を二重解放しない。

- [ ] **Step 1: 新規分先行 clone + 既存 ensureTotalCapacity + appendSliceAssumeCapacity + disarm**

上記 M6 手順を実装。`replaceLogCommits`（`model.zig:262-277`）の強例外パターンを踏襲。

- [ ] **Step 2: テスト拡充**

`test "appendLogCommits: 既存 items は再 dup せず所有権 move（perf phase1/M6）"`:
- 既存 2 件 + 新規 1 件で append。既存 items の ptr が move 後も同じ（再 dup で別 ptr にならないこと）を検証。
- `checkAllAllocationFailures` で OOM 時に既存 items 無傷・新規分（appended）解放・二重 free 無しを検証。
- paging シナリオ（100 → +100 → +100）で alloc 数が既存に比例しない（新規分のみ）を検証。
